## Current Positioning

The product is **Agentic Wallet Ops Center**. **The Desk** is the first app: a
live agentic trading desk that scans OKX opportunity surfaces, proposes wallet
actions, and gates every wallet-affecting action through a Black Box trace.

The **only durable competitive wedge** is *verifiable, policy-gated, agent
decision provenance* — a signed audit trail that a signing layer (wallet, smart
account, multi-sig, custody, agent runtime) can require *before* it signs. The
scanner, the UI, and the agent seats are all supporting evidence for that
wedge.

## Decisions Already Made

- Product framing is a trusted control plane for agentic wallet authority,
  not just an AI trading dashboard.
- The Black Box is the system of record for wallet-affecting decisions.
- Execution defaults to simulated / fixture-backed behavior.
- Live OKX calls are allowed for read-only evidence and scanning.
- Real funds are out of scope unless an explicit future gate enables a
  capped mode. *(No such gate exists today; see Invariants.)*
- The UI prioritizes live opportunity discovery and reviewable proposed
  actions over a replay-only dashboard.
- Secondary surfaces live in modals or focused views, not all on one long
  page.
- The next sprint theme is **"Prove the Black Box"** — server-side gate
  enforcement, ordered/veto-final verifier, and scanner-policy parity are
  the spine. Other work is framed around those three.
- Submission-day execution claims will be narrowed to `fixture` +
  `live-read`. `calldata`, `xlayer-testnet`, and `mainnet-capped` are cut
  from the policy UI until each has a real implementation path.

## Invariants (do not break without an ADR)

- Do not bypass `validateExecutionGate` for any append that lands in
  `blackbox/events.jsonl`. Browser code must round-trip through the local
  API for canonical appends.
- A `risk.verdict = veto` event is **final** for its ticket. No later
  `approved` overrides it. (To be enforced in the verifier — see sprint
  spine.)
- Required events (`candidate.created`, `risk.verdict`, `allocation.sized`,
  `route.quoted`, `user.confirmed`) must occur **strictly before**
  `execution.signed_or_simulated` in the ticket's event order — not merely
  exist somewhere in the chain.
- Malformed event payloads must fail **closed** (gate error), never throw
  out of the verifier.
- The scanner's `policy.allowed` verdict must use the same policy module
  that the Black Box verifier uses. No divergent policy logic.
- Do not let the agents directly mutate canonical wallet state without
  orchestrator validation.
- Do not print, log, or transmit `.env` contents anywhere — Black Box
  events, evidence files, console logs, demo artifacts.
- Do not claim mainnet execution unless it is actually implemented,
  capped, confirmed, and verified. The README, the policy dropdown, the
  agent docs, and the X-post draft must agree.
- Do not let a policy edit visually imply safety unless the corresponding
  canonical gate result updates after the edit.
- Every OKX live claim must carry either an evidence artifact or a
  visible source-health status. If the canary is stale, regenerate before
  demo.
- Default execution mode must remain `fixture`. Any change to
  `signingMode` or `executionMode` that broadcasts funds requires explicit
  human confirmation events and a real-funds-cap check that is implemented,
  not just declared.

## Open Decisions Before Next Sprint

(Each open question maps directly to one of Codex's seven outstanding
questions; resolutions belong in this file.)

1. **Veto override**: Is `risk.verdict = veto` truly final, or do we want
   an audited override role (e.g., a `risk.override` event signed by a
   second principal)? *Recommendation: final. No override for v1.*
2. **Browser appends**: Should browser-generated events ever land in
   `blackbox/events.jsonl`, or must every canonical append go through the
   local API + server verifier? *Recommendation: API-only. The browser
   may build event drafts but the server signs the canonical hash.*
3. **Policy edits**: persistent audited events, or temporary demo
   controls? *Recommendation: persistent `policy.updated` events with
   delta payload, signed by the API. Toggling integrity off should still
   record an event saying so.*
4. **Session anchoring**: what key signs the session hash? An ephemeral
   per-session ed25519 keypair generated by the local API at boot, with
   the pubkey exported into `web/public/data/integrity.json`? *Recommendation:
   yes for v1; revisit OKX Agentic Wallet co-signing later.*
5. **Authoritative OKX fields**: which OKX response fields are
   authoritative for quote, risk, wallet balance, and receipt proof, and
   how do we hash them into the execution event? *Open — Codex to
   propose during sprint.*
6. **Live execution modes for submission**: which modes are real for
   submission day? *Decision: `fixture` and `live-read` only. Cut the
   rest.*
7. **OKX wallet logged-out fallback**: what is the visible behavior when
   `onchainos wallet status` reports `loggedIn: false`? *Recommendation:
   show a banner; the Black Box still works (it is policy + provenance,
   not signing), but Mode pill becomes "logged-out / read-only".*
8. **Multi-chain scanner scope**: is non-Solana scanning in scope, or
   should the chain mapping and selector be removed? *Recommendation:
   Solana-only for the sprint; remove unused chain rows from `chainNames`
   in `src/opportunity-scanner.ts:87` or guard them.*
9. **Judge demo entry path**: radar-first or manual-console-first? README
   says radar-first; `demo/screenplay.md` still starts with the manual
   console. *Recommendation: radar-first; update screenplay.*
10. **Storage durability**: should `blackbox/events.jsonl` move to an
    append-only mode (e.g., open with `O_APPEND` only) and forbid
    `writeEvents` outside test setup? *Recommendation: yes, with a
    `eventsPath.test.jsonl` for tests.*

## Suggested Review Gates

### Pre-sprint gates (Claude Code reviews)

- Product coherence: every README claim has a code or test reference.
- UX clarity: the first-10-seconds flow (Radar → Stage → Confirm + simulate
  → Black Box → Replay) is unbroken on a fresh `npm run app`.
- Architecture: every wallet-affecting decision goes through one
  validation path, never two.

### Pre-sprint gates (Codex reviews)

- Implementation correctness: new verifier invariants are covered by
  adversarial tests (veto-then-approve, execution-before-confirmation,
  malformed payload, scanner-policy mismatch).
- Policy enforcement: scanner and Black Box import the same policy
  module; no duplicate inline policy logic.
- Regression risk: existing `submit:check` path remains green after the
  CI split (`verify:ci` no longer regenerates).

### No sprint gate passes without

- `npm run verify:ci` (new) passes — no artifact churn allowed.
- `npm run build:demo` (new) regenerates trace + scan + web bundle.
- Browser smoke test of the critical workflow (recorded as a GIF in
  `demo/` or a script-driven Playwright pass once added).
- Evidence files (`docs/evidence/*`) regenerated when live OKX behavior
  changes; otherwise dated at the top so staleness is visible.
- README updated when product claims change (modes table, agent roster,
  X-post draft).
